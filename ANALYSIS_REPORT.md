# Arak System Architecture Deep Analysis

## 1. Executive Summary
The Arak system is a modern, full-stack educational management platform. It utilizes a **React (Vite)** frontend and a **.NET 9** backend. The system is designed for scalability and maintainability, employing enterprise patterns such as the Repository Pattern and Layered Architecture.

## 2. Backend Architecture Details

### Layered Structure
- **DAL (Data Access Layer)**: Uses Entity Framework Core with SQL Server. Implements a `GenericRepository<T>` to provide standard CRUD operations, reducing boilerplate code across 20+ entities.
- **BLL (Business Logic Layer)**: Contains service implementations. Acts as a bridge between the API layer and data layer. Most services are currently passthroughs, allowing for future expansion of complex business rules.
- **PLL (Presentation Logical Layer)**: ASP.NET Core Web API controllers. Implements JWT-based authentication and role-based access control (RBAC).

### Key Observations & Best Practices
- **DTOS for Data Transfer**: The backend uses DTOs (Data Transfer Objects) to maintain a stable API contract with the frontend, preventing internal database structure changes from breaking the UI.
- **Async/Await**: The entire data pipeline is asynchronous, ensuring high performance and non-blocking I/O.
- **Dependency Injection**: Well-implemented DI for repositories and services, making the system highly testable.
- **Security**: Password hashing is handled by ASP.NET Identity (Bcrypt). JWT secrets are configurable via `appsettings.json`.

## 3. Frontend-Backend Integration

### API Synchronization
The frontend communicates with the backend via a centralized `api.js` service using **Axios**. 
- **Interceptors**: Automatically attach JWT tokens to every request and handle `401 Unauthorized` responses by redirecting to the login page.
- **Contract Mapping**: The backend explicitly handles naming differences (e.g., `StudentCode` in DB vs `StudentId` in UI) to ensure the frontend remains decoupled from the schema.

## 4. Stability & Security Audit

### Current Status
- ✅ **System Integrity**: Passed `dotnet build` with 0 errors.
- ✅ **Security**: RBAC is enforced on all sensitive endpoints (`[Authorize(Roles = "...")]`).
- ✅ **CORS**: Policy is restricted to known frontend origins (`localhost:5173`).

### Recommendations for Improvement
1. **UserId Consistency**: The `AuthService` currently returns a dummy `UserId = 0`. In a production release, this should map to the Identity User's GUID or a corresponding domain entity ID.
2. **Frontend Cleanup**: The `api.js` file contains legacy adapters for `json-server` (e.g., converting `_limit` to `_per_page`). These should be refactored to directly use the .NET API contract.
3. **Eager Loading**: Enhance performance by using `.Include()` in specific repository methods to avoid N+1 query issues in complex views (like Timetables).
